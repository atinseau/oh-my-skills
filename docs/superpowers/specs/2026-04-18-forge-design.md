# Forge — Generic Virtuous Development Cycle

**Status**: Released — v1.0 → v1.0.2
**Initial spec**: 2026-04-18
**Last revised**: 2026-04-20 (reflects v1.0.1 + v1.0.2 evolutions — see plans under `docs/superpowers/plans/`)
**Replaces**: `src/skills/swiftui-forge/` (breaking change, no migration)

## 1. Intent

Turn `swiftui-forge` into a **language- and framework-agnostic development framework** that:

- Works on any project regardless of stack or domain
- Has language-specific **profiles** that contribute extra rules/commands when a stack is recognized
- **Falls back** gracefully to a purely generic behaviour when no profile matches
- Establishes a **new norm**: a generic approach that maximizes LLM effectiveness on real projects

The framework prescribes **discipline**, not tooling. The agent chooses tools; the framework enforces the loop, the memory, the QA mindset, and the architectural guardrails.

## 2. Cardinal principles

1. The core prescribes **discipline** (7-step loop, memory, QA, TDD, max 5 iterations) — never specific tools.
2. Profiles contribute **high-level rules** (architecture, process, language paradigms) — never formatting or lint-territory concerns.
3. The agent **reasons and persists** its decisions (profile resolution, QA strategy, modules, bugs) inside `.forge/` — it does not re-decide each cycle.
4. **Universal fallback**: when no profile matches, the core alone works. The agent detects `build`/`test` commands itself.
5. **Continuity across sessions is the value prop**: forge costs more than it saves on day 1; break-even hits around day 3-4. It is overkill for one-shot scripts or throwaway repos. The skill's `description` frontmatter and SKILL.md intro state this explicitly so agents self-scope.

## 4. Per-project artefact: `.forge/`

Generated by bootstrap, maintained by the agent every cycle. Lowercase filenames throughout.

```
.forge/
├── index.md                          # Derived from all other files, regenerated at MEMORIZE
├── config.md                         # Profiles, commands, detection reasoning, last_consolidation
├── qa/                               # Agent's QA workspace (strategy + custom tooling)
│   ├── index.md                      # QA strategy (how to test, why, what tools)
│   └── <whatever the agent builds>   # scripts, fixtures, snapshots, utilities
├── architecture/
│   └── modules.md                    # Shared file: one `## <name>` block per module (seeded or enriched)
├── knowledge/
│   ├── pitfalls.md
│   └── dependencies.md
├── features/<name>.md
├── bugs/BUG-<NNN>.md
└── sessions/<date>-<topic>-<author-slug>.md
```

### `config.md` format

Markdown with YAML frontmatter. The frontmatter is structured state; the body is the agent's reasoning trail.

```markdown
---
profiles: [nextjs, typescript, react]
build_cmd: bun run build
test_cmd: bun test
lint_cmd: bun run lint
detected_at: 2026-04-18T00:00:00Z           # bootstrap time, never changes
last_consolidation: 2026-04-18T00:00:00Z    # updated at every MEMORIZE; drives desync detection
---

# Forge config — MyApp

## Why these profiles
- `nextjs` matched on `next.config.ts` — provides build/test/lint commands
- `typescript` matched on `tsconfig.json` — contributes rules
- `react` matched on `react` in package.json dependencies — contributes rules

## Decisions
- Took `build_cmd` from `nextjs` (only one proposing a build command)
- User confirmed monorepo root is `/apps/web` (asked during bootstrap)
```

### `index.md` format

Regenerated at MEMORIZE from the `.forge/` content. Short, dense, keyword-driven.

```markdown
---
updated: 2026-04-18
---

# Index

Profiles: [swiftui, typescript]
Last session: 2026-04-18-oauth2-login-arthur

## Modules
- auth — `src/auth/` — keywords: jwt, session, login
- api-client — `src/api/` — keywords: http, retry, timeout

## Open Bugs
- BUG-001 — login token not refreshed on expiry — keywords: jwt, session

## Features
- oauth2-login — status: wip — keywords: oauth2, login

## Pitfalls
- Token expiry not propagated to UI layer (see knowledge/pitfalls.md#token-expiry)

## Last Session
- 2026-04-18-oauth2-login-arthur — iterations: 3 — result: pass
```

### Session filenames

Sessions use the pattern `<date>-<topic>-<author-slug>.md`. `author-slug` is derived from `git config user.email` — take the part before `@`, lowercase, replace non-alphanumerics with `-`, truncate to 20 characters. If no git identity is configured, use `unknown`. This eliminates guaranteed git conflicts on shared branches.

### Memory rules

1. One file = one subject (bug, feature, session). `architecture/modules.md` is the single exception: a shared file holding one `## <name>` block per module.
2. Indexed-entity files carry `keywords:` in frontmatter — applies to `bugs/BUG-<NNN>.md` and `features/<name>.md`. Module blocks inside `modules.md` carry `keywords:` as a body line (seeded blocks omit until enrichment). Session files are narratives, not indexed entities, and carry `date`/`topic`/`author`/`iterations`/`result` in frontmatter but no `keywords`.
3. `index.md` is **derived**, never hand-edited; regenerated each MEMORIZE. Mid-cycle regeneration is allowed only for lazy module enrichment.
4. Sessions are **append-only and per-author** — one file per session per author.
5. `knowledge/pitfalls.md` and `knowledge/dependencies.md` are cumulative.
6. No duplication — pitfalls live once, referenced by links from modules/bugs.
7. QA artefacts do **not** live in the memory tree — they live in `.forge/qa/`.
8. Infrastructure pitfalls are recorded once — e.g. `no-test-suite` is added to `knowledge/pitfalls.md` the first time Step 4 hits Case B, never duplicated.

## 5. The loop (SKILL.md)

Seven steps. `SEE` renamed to `QA`. No step is ever skipped.

```
Step 0 — BOOTSTRAP
  If .forge/index.md is missing → execute references/project-bootstrap.md.
  Otherwise proceed to Step 1.

Step 1 — LOAD
  (a) Desync probe (prepended):
      - Read last_consolidation from .forge/config.md frontmatter.
      - git log --oneline --since=<last_consolidation> -- . ":(exclude).forge/"
      - git log --since=<last_consolidation> --name-only --pretty=format: -- <manifests>
      - If either non-empty: ask user to refresh (y/n).
        · yes → bounded refresh (deps, seed new modules, mark removed)
        · no  → note in `## Known staleness` section of upcoming session log
  (b) LOAD proper:
      - Read .forge/index.md; match task keywords against modules / bugs / features.
      - Read 2-3 relevant memory files + source of identified modules + last session log.
      - If a matched module is marked `seeded: true`, enrich it first (see Lazy Module Discovery).
      - Never scan the full codebase.

Step 2 — ACT
  Read references/tdd.md, references/architecture-guard.md, and every active
  profile's profile.md. Branch on task nature:
    - bug          → also read references/investigation.md
    - perf         → write a timed test first
    - adding tests to existing code → TDD red-first rule does not apply
    - dependency update → re-resolve config after edit; update knowledge/dependencies.md
  Write or update the test first. Implement in the identified module only.
  Run architecture-guard checks on every modified file.

Step 3 — BUILD
  Execute build_cmd from .forge/config.md frontmatter.
  0 errors → Step 4. Errors → back to Step 2.

Step 4 — TEST  (4 cases)
  Case A — test_cmd non-empty, runs normally.
           all pass → Step 5. failures → back to Step 2.
  Case B — test_cmd empty (or absent).
           Valid project state. JUDGE treats Tests = N/A. Record `no-test-suite`
           pitfall once at MEMORIZE. Do NOT propose framework install autonomously.
  Case C — test_cmd exits 0 with "no tests found" (jest --passWithNoTests etc.).
           Equivalent to Case A "All pass".
  Case D — test_cmd exits ≠ 0 with "no tests found" phrase AND no assertion/panic.
           Treat as Case B. When in doubt, prefer Case A (masked failure > misclassified empty suite).

Step 5 — QA  (never skipped)
  If .forge/qa/index.md is missing:
    - First-pass rule (eligibility gate): if .forge/sessions/ has zero entries
      with result: pass, a minimum-viable strategy is acceptable: 1 flow + 1
      criterion + explicit TODO (next cycle) markers for the other 6 discipline
      questions. Deleting qa/index.md does NOT re-enable the concession.
    - Otherwise: invest time and answer all 7 discipline questions.
    - Create .forge/qa/index.md and whatever scripts/fixtures are needed inside .forge/qa/.
    - Building the QA strategy is NOT counted as an iteration failure.
  If .forge/qa/index.md exists:
    - Follow the strategy. If insufficient, extend. Never bypass.
    - strategy: minimum-viable is valid for exactly one cycle. On the next
      cycle, all TODOs must be resolved or JUDGE fails.

Step 6 — JUDGE
  Evaluate: Build (0 errors), Tests (100% pass OR N/A for Case B/C), QA (objective pass).
  QA pass requires: criterion met AND no unresolved `TODO (next cycle)` markers from a prior cycle.
  All pass → Step 7. Any failure → back to Step 2.
  5th failure → stop, document in bugs/, notify user.

Step 7 — MEMORIZE (mandatory)
  1. Update architecture/modules.md if modules changed.
  2. Create/update features/<name>.md or bugs/BUG-<NNN>.md.
  3. Update qa/index.md if QA strategy evolved.
  4. Update knowledge/dependencies.md if dependencies changed.
  5. Append to sessions/<date>-<topic>-<author-slug>.md.
  6. Update last_consolidation in .forge/config.md frontmatter to now.
  7. Regenerate .forge/index.md from current .forge/ content.
```

## 6. Profiles

### Format

One markdown file per profile (v1): `profiles/<name>/profile.md`. A profile can later become a directory with sub-files if it grows.

```markdown
---
name: swiftui
description: Apple platforms (iOS, macOS, watchOS, tvOS) using SwiftUI and Swift Testing.
detect:
  files: [Package.swift, "*.xcodeproj", "*.xcworkspace"]
commands:
  build: xcodebuild build -scheme <scheme> -destination '<destination>'
  test: xcodebuild test -scheme <scheme> -destination '<destination>'
  lint: swiftlint
---

# SwiftUI profile

## When this profile activates
When any file from `detect.files` exists at the project root.

## Rules
### Architecture
- Protocol for every service; dependency injection via constructor
- No `@unchecked Sendable`
- `@MainActor` on ViewModels; `actor` for I/O services

### SwiftUI views
- Views = pure display, zero business logic
- Extract sub-views at 50+ lines
- `accessibilityIdentifier()` on all interactive elements
- `.task {}` modifier, not `onAppear`

### Error handling
- No force unwrap (`!`)
- No `try?` without justification
- Errors visible, never silent

### Tests
- Swift Testing framework preferred; existing XCTest acceptable
- Mocks via protocols

## Notes for the agent
- At bootstrap, detect the scheme and destination (macOS vs iOS Simulator) and
  persist resolved commands into .forge/config.md
- On dependency change, run `xcodebuild -resolvePackageDependencies` before BUILD
```

### Format rules

1. Frontmatter: `name`, `description`, `detect` are mandatory. `commands` is optional (a profile can contribute only rules).
2. Detection: file globs against project root. Optional `contents` matcher for content-based detection.
3. Command placeholders (`<scheme>`, `<destination>`, etc.) are resolved at bootstrap and persisted in `.forge/config.md`.
4. Body is free-form markdown. Rules are short actionable bullets — not essays.
5. Only **architecture & process** concerns — no formatting, no naming, no import order (lint territory).

### v1 profiles

- **`swiftui`** — content migrated from previous `swiftui-forge` `SKILL.md` Swift-specific rules.
- **`rust`** — Cargo-based projects. Detect on `Cargo.toml`. Commands: `cargo build`, `cargo test`, `cargo clippy -- -D warnings`. Rules: no `unwrap()` without justification, errors via `Result`/`thiserror`, modules ≤ 300 lines, clippy clean, integration tests under `tests/`.

### Multi-profile resolution

Multiple profiles can match. The agent arbitrates at bootstrap:

- 0 matches → generic mode; agent detects `build_cmd`/`test_cmd` from README/scripts/Makefile. If no test command can be inferred, leave `test_cmd` empty and let Step 4 Case B handle it.
- 1 match → take its commands.
- N matches:
  - If a single profile declares `commands`, take those; the others contribute rules.
  - If multiple declare `commands`, **ask the user** — no silent default.
- The agent writes its resolution reasoning in `.forge/config.md` under "Why these profiles" / "Decisions".

Invalidation: user deletes `.forge/config.md` to force a re-bootstrap. Automatic desync detection (see §8) catches out-of-band changes but does not invalidate the profile — only the user can do that.

## 7. Bootstrap (`references/project-bootstrap.md`)

Runs when `.forge/index.md` is missing.

```
1. SHALLOW SCAN
   - List root + one level. Read README, package.json/Cargo.toml/Package.swift/
     pyproject.toml/etc. if present.

2. PROFILE DETECTION
   - For each profiles/*/profile.md: parse `detect`, test patterns against the project.
   - Collect all matching profiles.

3. RESOLUTION (agent-driven)
   - Apply rules from §6 "Multi-profile resolution".
   - Resolve command placeholders by inspecting the project or asking the user.
   - Empty test_cmd / lint_cmd is acceptable (Step 4 Case B handles it).

4. ARCHITECTURAL SEED (minimal)
   - Do NOT try to understand every module up front.
   - List immediate subdirectories of the chosen root (src/, app/, lib/, Sources/, etc.).
   - For each, write one `## <name>` block with just `path:` and `seeded: true`.
   - No file reads. No role inference. No keywords.
   - Cap: >12 seed modules → keep first 12 by POSIX LC_ALL=C order, the rest go in `## other`.
   - Detailed module entries are filled lazily during LOAD (see §8).

5. WRITE .forge/
   - config.md (resolved frontmatter with last_consolidation = detected_at initially
     + "Why" section tracing decisions)
   - architecture/modules.md (seed blocks from Phase 4)
   - knowledge/pitfalls.md (empty, frontmatter only)
   - knowledge/dependencies.md (extracted from manifest)
   - index.md (regenerated from content above)

6. QA IS DEFERRED
   - Do NOT create .forge/qa/index.md at bootstrap.
   - The first Step 5 pass of the first cycle builds it (with eligibility for
     minimum-viable first pass — see §9).

7. RETURN TO CYCLE
   - Proceed to Step 1 (LOAD) of the current cycle.
```

Principles: the bootstrap **traces its reasoning** in `config.md`, and is **not silent** in case of ambiguity (asks rather than guesses).

## 8. Memory operations — lazy enrichment, desync, merge conflicts

Three operational disciplines attached to `.forge/` that complement the bootstrap seed:

### Lazy module discovery

Bootstrap writes seed entries (`name` + `path` + `seeded: true`). Full entries are built at LOAD when a task references a module:

1. Read the seed entry's `path` and list the files in it (one level deep).
2. Read 2-3 representative files (public entry, largest, or `index.*` / `main.*` / `mod.rs`).
3. Extract 3-6 keywords from filenames, top-level symbols, local README.
4. Rewrite the `## <name>` block body: drop `seeded: true`, add `role:`, `key_files:`, `keywords:` as body lines (not per-block frontmatter).
5. Regenerate `.forge/index.md`.

Budget: max 5 file reads per module per enrichment. Same cap applies to re-enrichment.

### Desync detection

`.forge/` assumes every code change passes through a cycle. Reality breaks this (git pulls, IDE renames, out-of-band `npm i`). Desync detection catches it at LOAD before keyword matches and bug lookups are corrupted.

- Trigger: every LOAD runs the probe described in §5 Step 1 (a).
- Refresh on accept: re-parse manifest dependencies (rewrite `dependencies.md` fully); seed new modules; mark removed modules with `status: removed` (keep entry for keyword history); update `last_consolidation`. Do NOT re-enrich existing non-seeded entries.
- Refresh on decline: record detected changes verbatim in a `## Known staleness` section of the session log. Proceed with known-stale memory.
- Not a replacement for MEMORIZE. Not automatic — the user always decides.

### Merge conflicts

`.forge/` is committed to git; concurrent work causes conflicts. Resolution by file type:

- `index.md` — derived. Discard both sides, regenerate from current `.forge/` content.
- `architecture/modules.md` — merge per `## <name>` block. Keep the more-enriched version. Regenerate index.
- `knowledge/pitfalls.md`, `knowledge/dependencies.md` — cumulative. Union both sides, deduplicate.
- `sessions/<date>-<topic>-<author-slug>.md` — per-author naming eliminates collisions by construction.
- `bugs/BUG-<NNN>.md`, `features/<name>.md` — genuine divergence; resolve by hand favouring the side with the richer Verification / Design decisions section.
- `config.md` — rare; take the later `last_consolidation`; for diverging commands, keep the side with `## Decisions` rationale.

## 9. QA runner (`references/qa-runner.md`)

Explains how the agent **builds** a QA strategy. Not a tool catalogue, not a template.

### Trigger

First Step 5 pass where `.forge/qa/index.md` does not exist. The agent must **invest time**: "what you build now will be used for months; understanding the app is cheaper than discovering regressions in prod."

### Minimum viable first pass (concession)

On the *very first* cycle after bootstrap (eligibility: `.forge/sessions/` has zero entries with `result: pass`), a minimum-viable `qa/index.md` is acceptable:

1. Eligibility gate: zero prior `result: pass` sessions. Deleting `qa/index.md` does NOT re-enable the concession.
2. Question 1 (app nature) answered concretely.
3. Exactly ONE primary flow defined with a concrete pass/fail criterion.
4. Tooling for that flow is present in `.forge/qa/` and has been run successfully **against the real target** (not a stub or dry run).
5. The remaining six discipline questions present as explicit `TODO (next cycle)` markers.

On the next cycle, every TODO must be resolved. `strategy: minimum-viable` is valid for exactly one cycle; rewriting TODOs to restart the clock is explicitly forbidden.

### Required discipline (7 questions)

The agent writes explicit answers in `.forge/qa/index.md`:

1. **App nature** — native UI, web UI, CLI, daemon, library, API, mixed? Determines everything else.
2. **Primary user journey** — concrete step sequence, not abstract.
3. **Host tools inventory** — exhaustive probe (POSIX shell example included; adapt for Windows/PowerShell).
4. **Adapted strategy** — visual (capture + diff), programmatic (calls + assertions), metric (perf/timing), or mixed.
5. **Tooling construction** — scripts under `.forge/qa/` that the agent creates, makes executable, tests. Carte blanche inside `.forge/qa/`.
6. **Objective pass/fail** — a written criterion per flow. Examples: "screenshot diff < 2%", "exit 0 + stdout matches X", "p95 < 500ms".
7. **Extensibility** — the strategy evolves. When a task doesn't fit, extend `qa/index.md` and the tooling; do not bypass.

### Recommended (not imposed) structure

```
.forge/qa/
├── index.md
├── flows/<name>.md
├── scripts/
├── fixtures/
└── snapshots/
```

### Anti-patterns

- Declaring `QA: pass` without objective verification
- Skipping QA because "unit tests cover it"
- Copying a strategy from another project without adapting
- Abstract generic strategy instead of project-concrete
- Writing `qa/index.md` in 2 minutes
- Letting `qa/index.md` go stale
- Using the minimum-viable concession on anything other than the very first pass
- Leaving `TODO (next cycle)` markers past the next cycle (JUDGE failure)

## 10. Universal code guidelines (in SKILL.md)

Active regardless of active profiles. Smallest common denominator across SwiftUI, Rust, Next.js, shell.

### Architecture
1. Max 300 lines per file
2. One file = one responsibility
3. Dependency injection via constructor/parameter — no hidden singleton
4. No god object (rough threshold: ~8 properties, ~10 public methods)
5. Module boundaries are explicit (interfaces, protocols, traits, types)

### Process
6. TDD: test first, always (except when adding tests to existing code — see ACT)
7. Errors visible, never silent
8. No empty `catch`, no error suppression without written justification
9. No hardcoded values for configurable things (magic numbers, URLs, timeouts)

### Tests
10. One test = one behaviour
11. Mocks via interfaces/protocols — no monkey-patching
12. Integration tests for pipelines and real I/O

### Memory (framework constraint)
13. Never scan the full codebase — `.forge/index.md` is the entry point
14. Save to `.forge/` after every successful cycle (MEMORIZE is never optional)
15. Regenerate `.forge/index.md` at every MEMORIZE

### QA (framework constraint)
16. QA is never skipped — every cycle hits Step 5
17. QA strategy belongs to the project (`.forge/qa/`), not to the profile
18. One **objective** pass/fail criterion per flow

### Iterations (framework constraint)
19. Max 5 iterations before stopping and documenting the blocker in `.forge/bugs/`
20. Every iteration produces new information — if two consecutive iterations fail on the same thing, change approach

## 11. Invocation scoping (pitch UX)

Forge's value is continuity across sessions. It is **overkill for one-shot scripts, single-file edits, throwaway repos**. On day 1 of a project, forge costs more than it saves. Break-even hits around day 3-4 of active work.

**Rule (in SKILL.md intro):** when bootstrapping a new project for a one-liner or a throwaway experiment, do NOT invoke forge — use the default Claude Code flow instead. Once `.forge/` already exists in a project, every task runs through the cycle — the memory is already paid for, and a one-line fix on day 30 still benefits from the discipline.

The frontmatter `description` reinforces this: "Multi-cycle development framework for work spanning multiple sessions. … overkill for one-shot scripts, single-file edits, or throwaway repos."

## 12. Out of scope for v1.x

- **Profile inheritance/composition** (`extends:`) — YAGNI, agent arbitrates instead.
- **Companion web profiles** (`nextjs`, `react`, `vue`, …) — separate spec once v1 stabilizes.
- **Automated tests of the skill itself** — consistent with repository convention (`CLAUDE.md`: "Skills are not unit tested").
- **Hook / CI integration** — skill remains purely agent-side.
- **Profile versioning / breaking-change handling** — not managed in v1.
- **Cache-hot references across cycles** (skip re-reading SKILL.md + references on cycle 2+). Requires reshaping skill consumption.
- **Fast-path for trivial tasks** — addressed indirectly by the §11 invocation scoping; not a separate mechanism.
- **QA-strategy periodic revisit rule** — requires defining "stale" for QA strategy; non-trivial.
- **Security guidance for session logs containing secrets** — separate reference, not a skill content change.

### Delivered in v1.x (was initially out of scope, now done)

- v1.0.1 — QA progressive first-pass concession; lazy module enrichment; empty `test_cmd` as project state (not failure).
- v1.0.2 — Desync detection at LOAD; per-author session filenames + merge conflict policy; scoped invocation + pitch UX.

## 13. Identified risks

- **Sloppy QA**: the biggest risk. Mitigation: `qa-runner.md` insists heavily on time investment, lists anti-patterns explicitly, MEMORIZE requires `qa/index.md` to be written for JUDGE to pass, and the minimum-viable first-pass concession is tied to an eligibility gate that cannot be reset by deleting `qa/index.md`.
- **Miscalibrated `rust` profile**: author is not a deep Rust expert. Mitigation: the implementation plan included a dedicated review step; v1.0.1 review softened the DI rule (trait objects / generics, not prescriptive `Arc<dyn Trait + Send + Sync>`).
- **Ambiguous multi-profile resolution**: the agent may hesitate. Mitigation: bootstrap asks the user rather than guessing silently.
- **Silent desync**: mitigated by the LOAD probe (v1.0.2). Declining refresh is tracked as `## Known staleness` in the session log, keeping the debt visible.
- **Multi-dev branch conflicts on `.forge/`**: mitigated by per-author session filenames (eliminates sessions/ conflicts) + documented resolution policy for index.md / modules.md / etc.

## 14. Success criteria

- `src/skills/forge/` exists with the layout from §3.
- `swiftui-forge` is deleted; installers and tests updated accordingly.
- Running the skill on an existing SwiftUI project produces a `.forge/` with the `swiftui` profile active and a `.forge/config.md` containing resolved commands + `last_consolidation`.
- Running the skill on a Rust project produces a valid `.forge/` with the `rust` profile active, and QA built via programmatic means (CLI assertions, integration tests) rather than visual.
- Running the skill on an unrecognized project (e.g. a Python CLI) falls back to the core alone, with the agent detecting `pytest`/`python -m <pkg>` itself (or leaving `test_cmd` empty if nothing is inferrable) and still producing a valid `.forge/`.
- LOAD's desync probe correctly surfaces out-of-band changes (git pulls, manifest edits) and offers the user a refresh, or records the staleness on decline.
- Two developers on the same branch on the same day produce two session files that do not conflict in git.
- The first cycle on a greenfield project can use minimum-viable QA; the second cycle on the same project cannot (eligibility gate enforced).
