# Forge Skill Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace `src/skills/swiftui-forge/` with a language-agnostic `src/skills/forge/` skill: a universal 7-step development loop, composable profiles (shipping `swiftui` and `rust`), self-built QA strategy, and a `.forge/` project memory layout.

**Architecture:** Markdown-only skill. Core (SKILL.md + `references/` + `templates/`) is universal. `profiles/<name>/profile.md` contributes detection patterns, commands, and high-level rules. The agent arbitrates profile resolution at bootstrap and persists decisions in `.forge/config.md`. Per-project QA strategy is built by the agent inside `.forge/qa/`.

**Tech Stack:** Pure markdown (no runtime code). Skill is discovered and installed by existing `scripts/install.sh` — no script changes expected. Validation happens via `bun check-types`, `bash -n` on scripts, and the existing testcontainers integration suite (`TESTCONTAINERS_RYUK_DISABLED=true bun test`).

**Spec reference:** `docs/superpowers/specs/2026-04-18-forge-design.md` — every task points to the section that defines its content.

**Execution notes:**
- This plan writes markdown content. There is no runtime to TDD. The "test" for each task is a structural read-back: does the file contain the required sections, frontmatter, and rules from the spec?
- The existing files under `src/skills/swiftui-forge/references/` provide the starting material for several tasks. Each task specifies whether to adapt or write fresh.
- Commits are grouped per task. Pre-commit hooks (`lefthook`) run `check-types`, biome, bash syntax check, and the full test suite. Expect ~60s on commit.
- The repo has no tests that reference `swiftui-forge` by name (verified by grep). Installer scripts discover skills dynamically from `src/skills/*/`. The rename is therefore self-contained to `src/skills/`.

## Task 3: Write `references/project-bootstrap.md`

**Spec:** §7 "Bootstrap".

**Starting material:** `src/skills/swiftui-forge/references/project-bootstrap.md`. **Rewrite mostly fresh** — the old file is SwiftUI-centric, the new flow is driven by profile detection.

**Files:**
- Create: `src/skills/forge/references/project-bootstrap.md`

- [ ] **Step 1: Write the file**

Structure (7 phases — matches spec §7):

1. `# Project Bootstrap` — intro: runs when `.forge/index.md` is missing.
2. `## Trigger` — orchestrator check: `.forge/index.md` exists? No → bootstrap.
3. `## Phase 1 — Shallow scan` — list root + one level; read `README`, `package.json`, `Cargo.toml`, `Package.swift`, `pyproject.toml`, `go.mod`, `Gemfile`, `pom.xml` if present. One `ls` + conditional reads. Do NOT deep-scan.
4. `## Phase 2 — Profile detection` — for each `profiles/*/profile.md` in the skill directory, parse the `detect:` frontmatter and test patterns against the project. Collect all matches. Pseudocode:
   ```
   matches = []
   for profile_dir in skills/forge/profiles/:
     fm = parse_frontmatter(profile_dir/profile.md)
     if any(glob(pattern, project_root) for pattern in fm.detect.files):
       matches.append(profile_dir.name)
   ```
5. `## Phase 3 — Resolution (agent-driven)` — state the rules from spec §6 "Multi-profile resolution":
   - 0 matches → generic mode: agent inspects `README`/`package.json` scripts/`Makefile` to propose `build_cmd`/`test_cmd`
   - 1 match → take its commands
   - N matches, 1 declares `commands` → take those; others contribute rules only
   - N matches, multiple declare `commands` → **ask the user** which profile owns the build. No silent default.
   - Resolve placeholders (`<scheme>`, `<destination>`, etc.) by inspecting the project or asking.
6. `## Phase 4 — Architectural scan` — identify primary modules (`src/`, `app/`, `lib/`, `Sources/`, `pkg/`, …). For each: name, path, role, key files, keywords. Write `.forge/architecture/modules.md` using `skills/forge/templates/module.md`.
7. `## Phase 5 — Write .forge/` — produce the files in this order:
   - `.forge/config.md` — frontmatter with resolved `profiles`, `build_cmd`, `test_cmd`, `lint_cmd`, `detected_at` (ISO date). Body: "Why these profiles" + "Decisions" sections tracing the reasoning.
   - `.forge/architecture/modules.md`
   - `.forge/knowledge/pitfalls.md` — empty body, frontmatter only (`type: pitfalls`, `updated: <date>`).
   - `.forge/knowledge/dependencies.md` — extracted from the project manifest. Each dependency: name, version, purpose (inferred or noted as unknown).
   - `.forge/index.md` — regenerated from the content above.
8. `## Phase 6 — QA is deferred` — explicit: do NOT create `.forge/qa/index.md` at bootstrap. Reason: a real task provides the context needed to design a meaningful QA strategy.
9. `## Phase 7 — Return to cycle` — proceed to Step 1 (LOAD) of the current cycle.
10. `## Principles` — two one-liners:
    - The bootstrap traces its reasoning in `config.md`.
    - The bootstrap is not silent in case of ambiguity — it asks.

Use concrete pseudocode or bash snippets where helpful, but keep language-agnostic. Do NOT reference `xcodebuild` or any Swift-specific tool.

## Task 8: Write `references/profile-authoring.md`

**Spec:** §6 "Profiles".

**Starting material:** None — write fresh. This file is the contract for contributors writing a new profile.

**Files:**
- Create: `src/skills/forge/references/profile-authoring.md`

- [ ] **Step 1: Write the file**

Structure:

1. `# Profile Authoring` — intro: a profile contributes detection patterns, optional commands, and high-level rules for a language or framework. Keep profiles short and focused.
2. `## Directory layout` — v1 layout: `profiles/<name>/profile.md` (single file). Profile can later become a directory with sub-files if content grows (e.g. `profiles/<name>/rules/concurrency.md`) — document that this is acceptable.
3. `## Frontmatter schema` — required fields:
   - `name` (string, must match directory name)
   - `description` (one-line)
   - `detect` — object with at minimum `files: [<glob>, <glob>]`. Optional `contents: [{file: <path>, matches: <regex>}]` for content-based detection.
   - `commands` — optional. Object with `build`, `test`, `lint` keys. Any can be omitted. Use `<placeholder>` syntax for values the agent must resolve at bootstrap.

   Example:
   ```yaml
   ---
   name: swiftui
   description: Apple platforms (iOS, macOS, watchOS, tvOS) using SwiftUI and Swift Testing.
   detect:
     files: [Package.swift, "*.xcodeproj", "*.xcworkspace"]
   commands:
     build: xcodebuild build -scheme <scheme> -destination '<destination>'
     test: xcodebuild test -scheme <scheme> -destination '<destination>'
     lint: swiftlint
   ---
   ```
4. `## Body content` — free-form markdown. Recommended sections:
   - `## When this profile activates` — one-sentence repeat of detection logic
   - `## Rules` — grouped by theme: architecture, concurrency, error handling, tests, language paradigms. Rules are **short, actionable bullets**, not essays.
   - `## Notes for the agent` — anything the agent needs to know at bootstrap or during cycles (e.g. how to resolve the `<scheme>` placeholder, post-dependency-update commands).
5. `## Rule scope — what belongs in a profile` — the cardinal rule: **architecture & process concerns only**. Explicitly listed:
   - ✅ Architecture (file size, SRP, DI, module boundaries)
   - ✅ Process (TDD patterns specific to the stack, error-handling idioms)
   - ✅ Language paradigms (concurrency model, ownership, type safety)
   - ❌ Formatting (indent, quotes, semicolons) — linter territory
   - ❌ Naming conventions — linter territory
   - ❌ Import order — linter territory
   - ❌ Highly tactical rules ("prefer map over for") — linter/review territory
6. `## Detection patterns` — practical guide:
   - File patterns: globs against project root
   - Content patterns: use when a file's presence is ambiguous (e.g. `package.json` exists for every JS project, but `next.config.*` is specific to Next.js)
7. `## Command placeholders` — list the pattern: `<placeholder>` syntax, resolved once at bootstrap and persisted in `.forge/config.md`. Example: `<scheme>` detected by `xcodebuild -list`.
8. `## Checklist for a new profile` — checklist a contributor runs before submitting:
   - [ ] Frontmatter is valid YAML
   - [ ] `detect.files` matches only projects of this type (tested on 2+ real projects)
   - [ ] `commands` placeholders are documented in "Notes for the agent"
   - [ ] Rules are architecture & process only (zero formatting rules)
   - [ ] Rules are short actionable bullets, not prose
   - [ ] Under 150 lines total (if larger, split into directory layout)

## Task 12: Write `profiles/rust/profile.md`

**Spec:** §6 "Profiles" — the rust v1 bullet-point outline.

**Starting material:** None. Write from scratch, following the swiftui profile as structural template.

**Files:**

- [ ] **Step 1: Write the profile**

Structure:

1. Frontmatter:
   ```yaml
   ---
   name: rust
   description: Rust projects using Cargo.
   detect:
     files: [Cargo.toml]
   commands:
     build: cargo build
     test: cargo test
     lint: cargo clippy -- -D warnings
   ---
   ```
2. `# Rust profile` — 1-line intro.
3. `## When this profile activates` — when `Cargo.toml` exists at project root (or in the active workspace member, if a monorepo).
4. `## Rules` sub-grouped:
   - `### Architecture`
     - Modules ≤ 300 lines (universal rule applies; restate for emphasis).
     - Public API goes in `lib.rs` or an explicit `pub mod` tree; no transitive `pub use` leaks.
     - No `unsafe` without a written safety comment immediately above the block, documenting invariants.
     - Crate boundaries mirror domain boundaries (split into sub-crates when a module has zero consumer coupling with the rest).
   - `### Error handling`
     - Errors via `Result<T, E>`; define error types with `thiserror` (or equivalent).
     - No `unwrap()` or `expect()` in production code without a written justification comment (panics acceptable in tests, examples, and `main.rs` if clearly labelled).
     - No panic in library code paths.
     - Error conversion via `From` / `?` — do not manually `.map_err(...)` when `From` can be implemented.
   - `### Concurrency`
     - Prefer `async` + `tokio` for I/O; `std::thread` only when async adds no value.
     - No `.await` inside a `Mutex` guard — use `tokio::sync::Mutex` for async contexts.
     - `Send + Sync` bounds on shared state are explicit, not inferred.
   - `### Tests`
     - Unit tests in `#[cfg(test)] mod tests { ... }` at the bottom of the source file.
     - Integration tests in `tests/` directory, one file per scenario.
     - Mocks via traits + dependency injection (constructors take `impl Trait` or `Box<dyn Trait>`).
     - `cargo test` must pass; `cargo clippy -- -D warnings` must be clean.
5. `## Notes for the agent`
   - No placeholders in commands — Cargo is self-contained. Persist commands as-is into `.forge/config.md`.
   - On dependency change (`Cargo.toml` edit), `cargo build` auto-fetches; no separate resolve command.
   - For workspaces (`[workspace]` in root `Cargo.toml`), detect members by parsing `workspace.members`. The agent may need to run `cargo <cmd> -p <member>` instead of top-level — ask the user at bootstrap if multiple members exist.
   - If `rustfmt.toml` exists, recognize `cargo fmt --check` as part of lint. Otherwise `cargo clippy` suffices.
