## Task 2: references/triggers.md

**Files:** Create `src/skills/forge/references/triggers.md`

**Rationale:** Triggers are the logic for when forge invokes itself. Writing them before `save.md` and `recall.md` because those two branch on which trigger fired.

### Content (write verbatim)

Structure:

1. `# Triggers` — intro: "Forge auto-invokes on these signals. For each, the agent decides whether to save, recall, or refresh memory."
2. `## Trigger matrix` — one-line summary table:
   | # | Signal | Action |
   |---|---|---|
   | 1 | First task in a codebase with no `.forge/` | Bootstrap |
   | 2 | Feature completed | Save feature + optional pattern/decision |
   | 3 | Reusable pattern surfaces | Save pattern |
   | 4 | Pitfall discovered | Save pitfall (+ bug if applicable) |
   | 5 | Memory may be stale | Desync probe + refresh ask |
   | 6 | User references past context | Recall |
3. `## Trigger 1 — Bootstrap` — detect + action (from spec §9):
   - Detect: user's first task in a directory; `.forge/index.md` missing.
   - Action: shallow scan (root + 1 level; read README + manifest); write initial `context.md` (language/framework/commands inferred from manifest); write empty `index.md` with just `updated:` frontmatter. Module files created lazily on first reference — do NOT pre-scan.
4. `## Trigger 2 — Feature completed` — detect + action:
   - Detect: user says "done", "ship it", "that's it"; or the agent determines the feature is implemented and verified.
   - Action: write `features/<name>.md`. Scan the work done for (a) a reusable pattern that justifies `knowledge/patterns.md` entry, (b) an architectural choice that justifies `knowledge/decisions.md` entry. Write those if applicable. Update `index.md`. Update `context.md:last_consolidation`.
5. `## Trigger 3 — Reusable pattern` — detect + action:
   - Detect: during coding, a solution shape will repeat (test idiom, error-handling shape, concurrency pattern, workflow).
   - Action: write or update `knowledge/patterns.md`. Deduplicate: if an existing entry covers the same pattern, update it rather than duplicate.
6. `## Trigger 4 — Pitfall` — detect + action:
   - Detect: during debugging or discovery, something is learned that should warn future work (breaking assumption, wrong default, subtle behaviour).
   - Action: write or update `knowledge/pitfalls.md`. If a bug was produced by the pitfall, also write `bugs/BUG-<NNN>.md` (see template).
7. `## Trigger 5 — Sync / desync` — detect + action:
   - Detect: at task start, run:
     ```bash
     git log --oneline --since="<last_consolidation>" -- . ":(exclude).forge/"
     git log --since="<last_consolidation>" --name-only --pretty=format: -- package.json Cargo.toml Package.swift pyproject.toml go.mod Gemfile
     ```
     (omit manifests absent from the project). If either non-empty, desync detected.
   - Also: if any file path in `modules/<name>.md` or `features/<name>.md` no longer exists on disk, desync detected.
   - Action: summarise changes to the user. Ask: *"Memory was last consolidated on `<last_consolidation>`. Refresh dependencies and module listings? (y/n)"*
     - Yes → bounded refresh: re-extract deps into `knowledge/dependencies.md`; stub new modules (create `modules/<name>.md` with `seeded: true`); mark removed modules (`status: removed` in frontmatter, keep file for keyword history). Update `last_consolidation` to now.
     - No → record `## Known staleness` block at the top of the current session log (or create one if nothing else memorable yet) listing detected changes. Proceed.
8. `## Trigger 6 — Recall` — detect + action:
   - Detect: user references past work ("how did we…", "what was the decision on…", "last time we did…"); or the agent needs context on an area where memory might apply.
   - Action: read `index.md`, keyword-match against user's task language, lazy-load 1-3 matching files. If a matched `modules/<name>.md` is `seeded: true`, enrich it first (see `recall.md`).
9. `## Not invoked for`:
   - One-shot scripts, throwaway edits, single-file fixes.
   - Routine tasks producing nothing memorable (typos, formatting, pure renames).
   - Projects where `.forge/` does not exist AND the user has not signalled they want memory.

## Task 3: references/save.md + references/recall.md

**Files:**
- Create: `src/skills/forge/references/save.md`
- Create: `src/skills/forge/references/recall.md`

**Rationale:** These are the two operations triggered by the 6 triggers. Save is the write path; recall is the read path. Written together because they reference each other.

### `save.md` content

1. `# Save` — intro: "How to write to `.forge/`. The goal is compact, deduped, keyword-indexed entries — not a diary."
2. `## Pick the template` — matrix:
   | Trigger | Template | File |
   |---|---|---|
   | Feature completed | `feature.md` | `features/<name>.md` |
   | Bug resolved | `bug.md` | `bugs/BUG-<NNN>.md` |
   | Pattern | `pattern.md` | appended entry in `knowledge/patterns.md` |
   | Pitfall | (inline) | appended entry in `knowledge/pitfalls.md` |
   | Decision | `decision.md` | appended entry in `knowledge/decisions.md` |
   | Session (memorable only) | `session.md` | `sessions/<date>-<topic>-<author-slug>.md` |
3. `## Compact discipline` — rules:
   - Target line counts: feature 15-30, bug 20-35, pattern 10-25, decision 15-30, session 10-25.
   - No redundant fields (no "What was asked" if Claude already has it from context).
   - Keywords always: 3-6 terms selected to match future recall queries.
   - Ruthless brevity on `## Learnings` bullets: one line each, no "also I realized that…" prose.
4. `## Deduplicate before save` — procedure:
   - For patterns and pitfalls: read the existing `knowledge/patterns.md` / `knowledge/pitfalls.md`; grep for overlapping keywords; if an entry already covers this, *update* rather than append.
   - For features and bugs: if an entry with the same `name`/`id` exists, update it.
5. `## When NOT to save` — explicit list (from spec §7):
   - Task produced no new learning (implemented exactly as Claude would have without forge).
   - Typo fix, formatting, rename, pure refactor with no new insight.
   - The "learning" is a generic programming observation, not project-specific.
   - If nothing is memorable, no `session.md` is written. Forge is not a diary.
6. `## After save`:
   - Update `index.md` (regenerate from directory content).
   - Update `last_consolidation` in `context.md`.
   - No commit — the user commits when they're ready.

### `recall.md` content

1. `# Recall` — intro: "How to read from `.forge/` on demand. Never scan the whole tree. `index.md` is always the entry point."
2. `## Entry point` — always `.forge/index.md`. Light by construction (max 100 lines).
3. `## Keyword match` — procedure:
   - Extract 3-5 salient terms from the user's task description (and their common variants/synonyms).
   - Scan `index.md` for matches against entry keywords.
   - Score: entries with ≥ 2 keyword matches are "high relevance"; 1 match is "low relevance"; 0 is skip.
4. `## Lazy load` — read the top 1-3 high-relevance files. If nothing matches strongly, do NOT over-read — proceed with the task using ambient context.
5. `## Lazy module enrichment` — when a matched `modules/<name>.md` is `seeded: true`:
   - Read `path` from frontmatter.
   - List the directory (one level deep).
   - Read 2-3 representative files: public entry point, largest file, or files matching `index.*` / `main.*` / `mod.rs` / `lib.rs`.
   - Extract role (one sentence), key_files (2-3 bullets), keywords (3-6).
   - Rewrite `modules/<name>.md` with enriched content.
   - Regenerate `index.md`.
   - Budget: max 5 file reads.
6. `## Cross-checks` — when task touches specific concerns, also load:
   - Testing concern → `knowledge/patterns.md` (test patterns)
   - Architecture decision → `knowledge/decisions.md`
   - Risky/sensitive area → `knowledge/pitfalls.md`
7. `## What recall does NOT do`:
   - Does not read the source tree proactively — only files identified via the index.
   - Does not re-enrich already-enriched module files (they're updated at save time, not read time).
   - Does not synthesise memory into prose summaries — returns the files as-is to Claude's context.

```markdown
# Forge

Project memory skill. Auto-invoked on specific signals to save or recall project context. Composes with superpowers — forge does NOT implement TDD, debugging, verification, or review. It remembers.

## What forge is for

Forge's value is continuity across sessions. Between your tasks and your breaks, `.forge/` preserves compact, keyword-indexed knowledge about your project — so the next session picks up with the right context automatically.

Best fit: projects spanning multiple sessions where learnings, patterns, and decisions accumulate. Overkill for one-shot scripts.

## When forge invokes itself

Six triggers. On each, forge reads or writes to `.forge/`.

| # | Signal | What happens |
|---|---|---|
| 1 | First task in a codebase with no `.forge/` | Bootstrap: write initial `context.md` + empty `index.md` |
| 2 | Feature completed | Save `features/<name>.md` (+ optional pattern / decision) |
| 3 | Reusable pattern surfaces | Append to `knowledge/patterns.md` |
| 4 | Pitfall discovered | Append to `knowledge/pitfalls.md` (+ bug file if fixed) |
| 5 | Memory may be stale (post-pull / manual edits) | Desync probe, ask user, bounded refresh or record staleness |
| 6 | User references past context | Recall: read `index.md`, keyword-match, lazy-load matching files |

See `references/triggers.md` for detection logic and detailed actions.

## Save

When triggers 2-4 fire: read `references/save.md`. Pick the right template, write compact content, dedup against existing entries, update `index.md`, update `last_consolidation` in `context.md`.

## Recall

When trigger 6 fires (or when a task may benefit from prior context): read `references/recall.md`. Always enter via `index.md`, keyword-match, lazy-load 1-3 matching files. Do not scan the memory tree.

## Lazy module enrichment

`modules/<name>.md` files are created as stubs (`seeded: true`) when a module is first referenced. On next recall, if the stub matches the task keywords, it is enriched in-place (read 2-3 files, extract role / key_files / keywords, rewrite). Budget: max 5 file reads per enrichment.

## Composition with superpowers

Forge is orthogonal to:
- `test-driven-development` — use this for TDD loops
- `systematic-debugging` — use this for bug investigation
- `verification-before-completion` — use this before claiming done
- `requesting-code-review` / `receiving-code-review` — use these for review discipline
- `writing-plans` / `executing-plans` / `subagent-driven-development` — use these for multi-step tasks

Forge does not replicate any of these. It remembers what they produce.

## Rules of forge

1. Auto-invoked — not always on.
2. Save only what's memorable — no session file for a no-op cycle.
3. Compact — target 10-30 lines per entry.
4. Keywords always — 3-6 terms per entry, for future recall.
5. Lazy — read `index.md` first, deeper files only on match.
6. Orthogonal — does not overlap superpowers skills.

## References

- `references/memory-structure.md` — `.forge/` layout, frontmatter schemas, index format, merge-conflict policy
- `references/triggers.md` — the 6 auto-invocation signals and their detection/action logic
- `references/save.md` — when/what/how to persist, dedup rules, when NOT to save
- `references/recall.md` — on-demand loading, keyword match, lazy enrichment

## Templates

Located at `skills/forge/templates/`:
- `feature.md` — completed feature
- `bug.md` — resolved bug
- `pattern.md` — reusable pattern (test idiom, archi shape, workflow)
- `decision.md` — architectural decision with rationale
- `session.md` — session log (only when memorable)
```
