# Setup -- Prerequisites & Config

**MANDATORY.** Run this before starting any mode. Do NOT skip steps 1-2 — they are hard blockers. If playwright-cli is missing, install it. Do NOT fall back to MCP tools, Playwright codegen, direct Playwright scripts, or any other browser automation tool.

## 1. Check playwright-cli binary (BLOCKING)

```bash
playwright-cli --version
```

- If the command exists and prints a version -> go to step 2
- If "command not found" -> **you MUST install it now**:
  ```bash
  npm install -g @playwright/cli@latest
  ```
  Then verify: `playwright-cli --version`
- **If installation fails** -> stop and tell the user. Do NOT proceed without playwright-cli. Do NOT use any alternative tool.

## 2. Check playwright-cli skill (BLOCKING)

The `playwright-cli` skill must be available (it provides the full command reference for browser automation).

- If the skill is listed in available skills -> go to step 3
- If the skill is NOT available -> **you MUST install it now**:
  ```bash
  playwright-cli install --skills
  ```
  This installs the skill files into `~/.claude/skills/` (the user's home directory, NOT the project directory). The skill must be globally available, not scoped to a single project.
  Then verify the skill appears in available skills.
- **If installation fails** -> stop and tell the user. Do NOT proceed without the playwright-cli skill.

## 3. Monorepo detection

Glob for `**/playwright.config.{ts,js}` from the project root, excluding `node_modules`.

- **One config found:** Set the working context to that config's directory. Proceed to step 4.
- **Multiple configs found:** This is a monorepo with multiple apps. List each app with its relative path, for example:
  ```
  Found 3 Playwright configs:
    1. apps/web          (apps/web/playwright.config.ts)
    2. apps/admin        (apps/admin/playwright.config.ts)
    3. packages/ui       (packages/ui/playwright.config.ts)
  ```
  Ask the user which app they want to work on. Set the working context to that app's directory. All subsequent paths (config.yaml, test_dirs, discovery_root) are relative to this app directory.
- **No config found:** Ask the user where their Playwright config lives. If they don't have one, this skill cannot proceed.

## 4. Load or create config.yaml

### If config.yaml exists

1. Read `.discovery/config.yaml`
2. **Check for legacy format:** If the config is missing the `version` field, or contains a `test_dir` field (singular) or a `spec_suffix` field, stop and trigger **healthcheck mode** -- do not attempt to migrate the config inline. Healthcheck will handle the migration.
3. Read `.discovery/scenarios/_index.md` -- understand current coverage
4. If the user gives a specific task -> go to that mode
5. If the user says "continue" or "what's next?" -> suggest the next step based on coverage gaps
6. Default mode is **explore** -- if unsure, explore

### If no config.yaml (first run)

1. Read the Playwright config file detected in step 3 (`playwright.config.ts` or `.js`) -> extract `webServer.command`, `webServer.url`, `testDir`
2. Scan subdirectories of the detected `testDir` to build the `test_dirs` map. Each immediate subdirectory becomes a domain entry. For example, if `testDir` is `tests/e2e` and it contains `auth/`, `dashboard/`, `settings/`:
   ```yaml
   test_dirs:
     auth: tests/e2e/auth
     dashboard: tests/e2e/dashboard
     settings: tests/e2e/settings
   ```
   If `testDir` has no subdirectories (flat structure), create a single entry using the directory name as the key:
   ```yaml
   test_dirs:
     e2e: tests/e2e
   ```
3. Glob for existing patterns -> `helpers/`, `mocks/` or `fixtures/`
4. Propose the detected values to the user for confirmation
5. Write `.discovery/config.yaml`
6. If a value cannot be detected -> ask the user

### config.yaml format

```yaml
# Auto-generated by nightwatch -- edit freely
version: 1
discovery_root: tests/.discovery
dev_command: pnpm e2e:dev
base_url: http://localhost:5173
test_command: pnpm e2e
test_dirs:
  auth: tests/e2e/auth
  dashboard: tests/e2e/dashboard
helpers_dir: tests/helpers
mocks_dir: tests/mocks/data
```

## 5. Session bootstrap auto-correction

After loading config and reading `_index.md`, perform silent coherence checks before entering any mode. Fix everything automatically. Only report blocking issues to the user.

### Dead link removal

For each scenario referenced in `_index.md`, check that the linked file exists on disk. If the file does not exist, remove the row from the index. This is silent -- do not ask the user.

### Stale scenario detection

For each scenario in the index:
1. Read the scenario file to find its `Page` field
2. Read the corresponding map file and check its `Last explored` date
3. If the page was re-explored after the scenario was last validated (i.e., the map's `Last explored` is more recent than the scenario's last test run), the page content may have changed. Mark the scenario status as `stale` in the index.

This is a page-level check -- if any scenario's page was re-explored, all scenarios for that page are potentially stale.

### Coverage recalculation

Recalculate the `## Coverage` table in `_index.md` by counting actual scenario statuses per domain:
- Count scenarios, covered scenarios, and pages per domain
- Update the `Last explored` column from the most recent map file in that domain

Overwrite the table with accurate numbers.

### Structural coherence check

If any of the following are detected, stop and trigger **healthcheck mode** before continuing:
- Unknown fields in `config.yaml` that are not part of the schema
- Scenario files that do not follow the expected format (missing required fields like `Status`, `Page`, `Steps`)
- `_index.md` structure that does not match the expected format (missing `## Coverage` table, missing domain sections)

Everything else (dead links, stale markers, coverage counts) is fixed silently. Only structural incoherence blocks the session.
